# PRD: Product, Inventory, Shipping, and Settings UX Reset

## 한 줄 목표
Seleccase Inventory를 `상품 관리 + 테이블 중심 재고 운영 + 설정 내부 스토어 연결 + 업로드 직후 분류 가능한 운송장 미리보기` 흐름으로 재정리한다.

## 현재 문제
- `상품 관리`와 `기준 데이터`라는 표현이 섞여 있어 상품/창고 기준정보의 주인이 무엇인지 한눈에 읽히지 않는다.
- `src/app/components/inventory/InventoryWorkspace.tsx`의 개요 화면은 창고 선택 외에 실질적인 컬럼 필터, 검색, 컬럼 가시성 제어가 부족해 재고 표를 작업 표면으로 쓰기 어렵다.
- `src/app/(protected)/inout/InOutForm.tsx`는 입고/출고 버튼으로 진입한 뒤에도 다시 입고/출고를 고르게 하고, `표 붙여넣기` 패널까지 포함해 팝업의 섹션 수가 과도하다.
- 재고 운영 화면은 헤더, 탭, CTA, 보조 배지, 탭 내부 설명이 동시에 쌓여 실제 표보다 chrome이 먼저 보인다.
- `src/app/(protected)/shipping/ShippingView.tsx`는 업로드 화면 안에 `연동 준비 상태`, 채널별 미연결 카드, 이동 버튼을 반복해 같은 맥락을 여러 번 설명한다.
- 운송장 업로드 뒤에는 원본 엑셀 행만 보여주고 있어, 사용자가 어떤 행이 네이버/쿠팡으로 분류됐는지 즉시 파악하기 어렵다.
- `src/app/(protected)/integrations/IntegrationsView.tsx`와 `src/app/(protected)/settings/SettingsView.tsx`가 스토어 연결 책임을 나눠 갖고 있어 설정 구조가 겹친다.
- 상단 제목/서브타이틀, 설명 카드, 안내용 섹션이 실제 작업 표면보다 커서 AI slop처럼 느껴지는 빈 여백과 중복 copy가 생긴다.
- `기준 데이터`와 `/integrations`는 더 이상 canonical source of truth가 아니어야 하는데, 현재 문서와 화면 곳곳에서 아직 주인처럼 보인다.

## 사용자
- 상품/창고 관리자: 상품과 창고 기준정보를 빠르게 찾고, 이름과 상태를 직접 수정해야 한다.
- 창고 관리자: 창고별 재고를 표에서 바로 보고 상품명, 창고, 상태 기준으로 빠르게 걸러야 한다.
- 운영 매니저: 입고/출고를 최소 입력으로 다건 처리하고, 복잡한 보조 섹션 없이 결과만 빨리 확인하고 싶다.
- 배송 담당자: 엑셀 업로드 직후 어떤 행이 네이버/쿠팡 주문과 매칭됐는지 배지와 필터로 확인하고 싶다.
- 관리자: 설정 화면 안에서 네이버/쿠팡 연결 상태를 보고 연결/변경을 처리하고 싶다.

## 핵심 재설계 판단
- `상품 관리`는 상품/창고 기준정보의 top-level domain이다.
- `재고 운영`은 목록과 이력 view, 입고/출고 action 중심의 운영 허브다.
- `설정`은 `스토어 연결`의 canonical owner다.
- `운송장`은 연결 설명 페이지가 아니라 `업로드 → 미리보기 → 분류 → 매칭/발송` 실행 화면이다.
- `기준 데이터`는 top-level destination이 아니라 `상품 관리`의 과거 표현으로만 남는다.
- `/integrations`는 별도 owner가 아니라 `설정 > 스토어 연결`로 보내는 legacy alias다.
- 중복 설명 카드와 장식용 안내 섹션은 제거하고, 실제 작업 표면을 먼저 보여준다.

## 목표 정보 구조
```text
대시보드
상품 관리
├── 상품
└── 창고
재고 운영
소싱
├── 외부 공장
└── 입고 예정
운송장
분석
설정
└── 스토어 연결
```

## 메뉴 원칙
- 1차 메뉴는 직접 가치가 있는 목적지여야 한다.
- 확장형 메뉴는 2개 이상 의미 있는 child screen이 있을 때만 사용한다.
- `상품 관리`는 top-level direct item으로 두고, 내부에서 `상품`과 `창고`를 구분한다.
- `소싱`은 확장형을 유지한다.
- `재고 운영`, `운송장`, `분석`, `설정`은 direct item으로 유지한다.
- `스토어 연결`은 도메인 이름으로 유지하되 `설정` 안의 canonical section 또는 child screen으로 둔다.

## 핵심 시나리오
1. 사용자는 `상품 관리`로 이동해 상품과 창고 기준정보를 수정한다.
2. 사용자는 `재고 운영`으로 이동한다.
3. 상단 툴바에서 창고, 상품명, 상태, 옵션, 컬럼 표시를 조정하며 재고 표를 바로 필터링한다.
4. `입고` 또는 `출고`를 누르면 고정 타입의 빠른 입력 팝업이 열리고, 최소 필드만 포함된 편집 테이블에서 다건 입력한다.
5. 복잡도가 높아지면 `CSV`와 `이력`은 `재고 운영` 하위 페이지에서 별도로 다룬다.
6. 사용자는 `설정`에서 네이버/쿠팡 연결 상태를 보고 연결하거나 값을 변경한다.
7. 사용자는 `운송장`에서 엑셀을 업로드한다.
8. 시스템은 연결된 스토어에서 주문을 조회하고 이름/주소를 비교해 업로드 행을 `네이버`, `쿠팡`, `미분류`, 필요 시 `중복 후보`로 분류한다.
9. 사용자는 분류 배지와 필터를 이용해 미분류 행만 확인하거나 채널별로 발송 대상을 정리한다.

## 기능 요구사항

### 0. 상품 관리
- `상품 관리`는 상품과 창고 기준정보의 top-level owner다.
- 이 화면은 다음 두 책임을 분리해 제공한다.
  - `상품`: SKU, 옵션, 상태, 상품명 등 실제 재고 운영에서 참조되는 상품 속성
  - `창고`: 창고명, 사용 여부, 식별 정보, 운영 메모 등 재고가 머무는 장소 속성
- `상품 관리`는 `재고 운영`의 조회 대상 데이터를 제공하지만, 재고 수량 자체를 소유하지는 않는다.
- `기준 데이터`는 내부 설명이 아니라 `상품 관리`를 가리키는 과거 표현으로만 취급한다.

### 1. 재고 운영 허브
- `재고 운영`은 단일 first-level hub로 유지한다.
- 기본 워크스페이스는 다음을 가진다.
  - `목록`
  - `이력`
  - `입고`
  - `출고`
  - `CSV`
- 단, `CSV`와 `이력`이 한 화면 UX를 과도하게 복잡하게 만들면 `/inventory/csv`, `/inventory/history` 같은 하위 페이지로 분리할 수 있다.
- 창고 컨텍스트는 허브 공통 상태로 유지한다.

### 2. 재고 목록 테이블
- 목록의 기본 레이아웃은 `compact filter bar + dense table`이어야 한다.
- 표는 최소 다음 정보를 제공한다.
  - 상품명
  - 옵션
  - 창고
  - 현재 재고
  - 최근 입고
  - 최근 출고
  - 상태 배지
- 최소 다음 필터를 지원한다.
  - 창고 dropdown
  - 상품명 검색
  - 상태 필터
  - 필요 시 옵션/재고 상태 보조 필터
- 컬럼 표시/숨김 dropdown을 지원한다.
- 행 애니메이션은 표 가독성을 해치지 않는 미세한 수준으로 제한한다.

### 3. 빠른 입고/출고
- 상단 `입고` 버튼은 입고 전용 팝업만 연다.
- 상단 `출고` 버튼은 출고 전용 팝업만 연다.
- 팝업 안에 다시 `입고/출고` 전환 버튼을 두지 않는다.
- 팝업은 최소 섹션만 가진다.
  - 제목
  - 공통 컨텍스트(창고, 날짜)
  - 편집 테이블
  - 검증/저장 footer
- 기본 입력 방식은 compact editable table이다.
- 유지할 기능:
  - 행 추가
  - 행 삭제
  - 마지막 행 복제
  - inline validation
- 제거할 기능:
  - `표 붙여넣기`
  - 팝업 안의 중복 타입 전환
  - 설명만 하는 보조 카드

### 4. CSV와 이력의 분리 기준
- `CSV`와 `이력`은 기본적으로 허브 내부에서 시작한다.
- 다음 중 하나가 발생하면 재고 운영 하위 페이지로 분리한다.
  - 툴바와 표가 동시에 과밀해져 같은 CTA가 반복된다.
  - 감사성 메타데이터 때문에 목록/이력 표 가독성이 무너진다.
  - 모바일에서 한 화면에 유지하기 어렵다.

### 5. 설정 안의 스토어 연결
- `설정` 화면은 `스토어 연결`의 canonical owner가 된다.
- 네이버/쿠팡 각각에 대해 다음을 표시한다.
  - 연결 여부
  - 마스킹된 요약
  - 최근 변경 시각
  - `연결` 또는 `변경` 버튼
- provider별 상태 요약과 편집 form은 가능한 한 같은 surface 안에 둔다.
- provider별 deep link 또는 section focus를 지원한다.
- `/integrations`는 새 owner가 아니라 `설정 > 스토어 연결`로 보내는 alias/redirect 후보로 간주한다.

### 6. 운송장 업로드와 분류
- `운송장` 화면에서 `연동 준비 상태` 같은 별도 설명 섹션은 제거한다.
- 업로드 섹션 아래의 첫 번째 표는 원본 엑셀 행이 아니라 `분류된 업로드 미리보기`가 되어야 한다.
- 네이버가 연결되어 있으면 네이버 주문을 조회해 이름/주소 비교로 해당 row를 분류한다.
- 쿠팡이 연결되어 있으면 쿠팡 주문을 조회해 이름/주소 비교로 해당 row를 분류한다.
- 분류 결과는 badge로 표시한다.
  - `네이버`
  - `쿠팡`
  - `미분류`
  - 필요 시 `중복 후보`
- 사용자는 최소 다음 필터를 사용할 수 있어야 한다.
  - 전체
  - 네이버
  - 쿠팡
  - 미분류
- 연결되지 않은 provider는 비활성 상태 설명 대신 활성 버튼으로 노출하고, 클릭 시 해당 provider 설정 section으로 이동시킨다.

### 7. 운송장 핵심 흐름 유지
- 기존 `업로드 → 미리보기 → 매칭/발송` 흐름은 유지한다.
- 분류 표는 발송 대상 정리에 필요한 1차 surface가 된다.
- 채널별 발송 액션은 분류 결과를 기반으로 계산한다.
- 채널별 발송 액션은 분류 표의 header/footer action rail에 붙이는 편을 우선한다.

## 비기능 요구사항
- 기존 `inventory`와 `transactions` 원장 구조는 유지한다.
- 기존 사용자 스코프와 RLS 가정은 유지한다.
- 표와 필터는 desktop 우선 dense layout을 쓰되, 모바일에서는 우선순위 컬럼만 남기거나 stacked layout으로 축소한다.
- body text는 14px 이상, 탭/버튼/입력은 44px 이상을 유지한다.
- 상단 타이틀과 설명은 최소화하고, 같은 맥락을 카드/배지/문장으로 반복하지 않는다.

## MVP 제외 사항
- 범용 커넥터 프레임워크
- 바코드/스캐너 연동
- 자동 재주문/추천 엔진
- 창고 간 이동 자동화
- 승인 워크플로우

## 성공 기준
- `상품 관리`가 top-level domain으로 보이고, 상품과 창고 책임이 분리되어 있다.
- 재고 표에서 창고와 상품명 기준 필터링이 한 화면 안에서 즉시 가능하다.
- 입고/출고 팝업은 고정 타입이며, 표 붙여넣기 없이도 다건 입력이 빠르다.
- `스토어 연결` 편집 위치가 `설정`으로 단일화된다.
- 운송장 업로드 직후 각 row의 채널 분류가 배지로 보이고, 미분류만 따로 볼 수 있다.
- 불필요한 타이틀/서브타이틀/설명 카드가 제거되어 실제 표와 액션이 먼저 보인다.
